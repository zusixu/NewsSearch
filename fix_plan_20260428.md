### T1: web_access_transport.py 日志补全

**问题**：805 行网络 I/O 代码零日志输出，15+ 处静默吞异常（`except Exception: pass` / `return None`），违反 `rule.md` §6.1（所有网络 I/O 必须日志）和 §6.2（禁止 `except Exception: pass`）。

**修复范围**：
- 添加 `from app.logger import get_logger; logger = get_logger(__name__)`
- CDPProxyClient 所有方法：记录请求/响应/异常（`is_available`, `new_tab`, `eval_js`, `navigate`, `close_tab`, `get_page_text`, `search_bing`）
- WebAccessTransport 所有方法：记录搜索查询、抓取 URL、降级决策、响应耗时
- 所有 `except Exception: pass` / `return None` → `logger.exception()` 或 `logger.warning()` + 返回
- `time.sleep(2)` / `time.sleep(3)` (行176, 189) → 加 `logger.debug` 记录等待原因，后续考虑可配置化

**规范遵循**：
- `rule.md` §6.1: 所有网络 I/O 必须记录请求开始、响应状态码、耗时
- `rule.md` §6.2: 禁止 `except Exception: pass`
- `rule.md` §6.3: 使用 `app/logger/` NDJSON 格式，`logging.getLogger(__name__)`

**不改**：不重构类结构、不改变方法签名、不添加重试逻辑（最小变更原则）。

### T3: 存储层分层修复 + ChatMessage/PromptRenderer 迁移

**问题 #8**：`app/storage/database.py` 导入 `app/analysis/adapters/contracts` 中的 `PromptTaskType` 和 `app/analysis/prompts/profile` 中的 `PromptProfileConfig`、`PromptProfileLoader`、`TaskTemplateMapping`，违反分层原则（storage 不得依赖 analysis）。

**问题 #12**：`ChatMessage` 和 `PromptRenderer` 定义在 `github_models.py`，但被 `openai_compatible.py`、`engine.py`、`react/engine.py` 依赖，位置不当。

**修复方案**：

分两步走，先迁 ChatMessage/PromptRenderer，再解 storage 依赖：

**步骤 1 — 迁移 ChatMessage/PromptRenderer**：
- 将 `ChatMessage` dataclass 从 `github_models.py:64-77` 移到 `contracts.py`
- 将 `PromptRenderer` Protocol 从 `github_models.py:136-157` 移到 `contracts.py`
- 更新 `github_models.py` 中的导入：`from app.analysis.adapters.contracts import ChatMessage, PromptRenderer`
- 更新 `openai_compatible.py:41`、`engine.py:31`、`react/engine.py:44` 的导入路径
- `github_models.py` 中保留 re-export（`__all__`）避免破坏外部引用

**步骤 2 — 解除 storage→analysis 依赖**：
- 在 `app/storage/` 下新建 `types.py`，定义存储层自用的 `PromptTaskType` 和 `PromptProfileConfig` 的 storage 专用变体
- 或者将 `PromptTaskType` 上提至独立 `app/types/` 或 `app/storage/types.py`（需与 analysis 层共用同一枚举）
- 推荐：将 `PromptTaskType` 移到 `app/storage/types.py`，analysis 层改为从 storage 层导入（storage 是更底层模块，analysis 依赖 storage 是合理的方向）
- `PromptProfileConfig` 和 `TaskTemplateMapping` 同理
- `PromptProfileLoader` 留在 analysis 层，storage 层改为只接收序列化后的数据

**规范遵循**：
- `rule.md` §3.1: 下层不得依赖上层，storage 不得依赖 analysis
- `rule.md` §3.3: 适配器间共享类型应在 contracts.py 或独立 types.py
- 最小变更：迁移后保持所有公共 API 不变

### T4: ReAct 引擎 duck-typing 修复

**问题**：`app/analysis/react/engine.py:578-634` 的 `_call_llm_raw` 方法通过 duck-typing 访问适配器的私有方法 `_render_messages`、`_build_payload`、`_read_token`/`_read_api_key`、`_post`，这些方法不在 `AnalysisAdapter` Protocol 中。

**修复方案**：

**方案 A（推荐）— 在 Protocol 中添加低级接口**：
- 在 `AnalysisAdapter` Protocol 中添加 `analyse_raw(analysis_input: AnalysisInput) -> dict[str, Any]` 方法
- 该方法返回原始 API 响应 dict（当前 `_call_llm_raw` 的返回值）
- 两个适配器实现该方法
- ReAct engine 改为调用 `self.adapter.analyse_raw()` 而非 duck-typing 私有方法
- 好处：接口稳定，新增适配器只需实现 Protocol

**方案 B — 提取 LLMTransport Protocol**：
- 新建 `LLMTransport` Protocol 定义 `render_messages` / `build_payload` / `post` 公共方法
- 适配器实现 `LLMTransport`
- ReAct engine 依赖 `LLMTransport` 而非 `AnalysisAdapter`

选方案 A，更简单且符合最小变更原则。

**规范遵循**：
- `rule.md` §3.3: 适配器通过 Protocol 定义接口，不依赖私有方法
- `rule.md` §1.1: 最小变更

### T6: 关系类型覆盖级联修复

**问题**：`app/chains/candidate_generation.py` 三阶段流水线中，每阶段完全覆盖 `ChainNode.relation_to_prev`，导致最终所有节点只剩 `UPSTREAM_DOWNSTREAM`，丢失 `SAME_TOPIC` 和 `TEMPORAL` 语义信息。

**覆盖级联**：
```
Stage 1 (same_topic_grouping.py:110) → SAME_TOPIC
Stage 2 (temporal_connection.py:73)   → TEMPORAL (覆盖 SAME_TOPIC)
Stage 3 (upstream_downstream.py:143)  → UPSTREAM_DOWNSTREAM (覆盖 TEMPORAL)
```

**修复方案**：

**推荐方案 — 选择最强语义关系**：
- 每个阶段只在关系"更强"时覆盖，保留较弱关系
- 语义强度：`UPSTREAM_DOWNSTREAM` > `TEMPORAL` > `SAME_TOPIC`
- 即：如果 Stage 3 判定两个节点间存在上下游关系，则覆盖；若无法判定，保留 Stage 2 的 TEMPORAL；若 Stage 2 也无法判定，保留 Stage 1 的 SAME_TOPIC
- 在 `ChainNode` 上无需新增字段，只需修改各阶段不盲目覆盖

**实现细节**：
- `temporal_connection.py` `_reorder_chain`: 仅在重排确实改变了顺序时设置 `TEMPORAL`，否则保留原 `relation_to_prev`
- `upstream_downstream.py` `_reorder_chain`: 仅在确认上下游映射成功时设置 `UPSTREAM_DOWNSTREAM`，否则保留原值
- 更新对应测试：`test_temporal_connection.py`、`test_upstream_downstream.py`

**规范遵循**：
- `rule.md` §1.2: 数据不可变（ChainNode frozen=True，使用 `dataclasses.replace`）
- `rule.md` §5.3: 修改后测试覆盖
